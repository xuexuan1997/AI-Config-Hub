# ADR-0001：采用 TypeScript pnpm 模块化单体

## 状态

已接受，适用于 MVP 技术基线。

## 背景

AI Config Hub 需要同时提供 Electron 桌面端和无图形环境 CLI，并共享配置发现、解析、生效计算、诊断、转换、部署和回滚能力。首版需要支持四种工具与四类资源，还需在 Windows、macOS 和 glibc 2.28 Linux 上可靠构建和分发。

如果过早拆分为多个本地服务，安装、进程管理、调试和版本协商会放大 MVP 风险；如果直接允许动态加载第三方适配器，又会引入代码执行、签名、沙箱和兼容协商问题。

## 决策

采用全 TypeScript 的 pnpm workspace 模块化 Monorepo：

- `apps/desktop` 和 `apps/cli` 是两个入口。
- 领域规则、用例、适配器、扫描、部署、存储、Git、API 和共享类型以独立包组织。
- 各包只通过公开入口依赖，禁止跨包深层导入。
- 工具适配器通过静态注册表在编译时接入。
- CLI 与 Electron 主进程直接复用同一核心用例。

详细边界见[系统架构](../architecture/overview.md)与[工具适配器](../architecture/adapter-system.md)。

## 备选方案

### 动态插件微内核

扩展灵活，但 MVP 必须额外解决插件来源、签名、权限、沙箱、API 兼容和升级失败恢复，当前收益不足以覆盖成本。

### 多进程本地服务

隔离性较强，但需要进程发现、端口或套接字管理、协议版本、服务生命周期和多产物升级，增加桌面安装和受限 Linux 环境的运维复杂度。

### 单包应用

初始文件较少，但会让桌面、CLI、适配器和基础设施边界混杂，难以进行独立测试和未来拆分。

## 后果

正面影响：

- 桌面端与 CLI 的行为由同一实现保证。
- 单一依赖图、锁文件和构建流程降低发布复杂度。
- 包边界支持独立测试，并为未来插件化提供接口基础。

负面影响：

- 所有包共享发布仓库，必须用依赖规则防止无序耦合。
- 适配器更新需要重新构建和发布应用。
- 静态注册无法满足未经审核的第三方适配器即时安装。

## 复审条件

出现以下任一情况时复审本决策：

- 适配器数量或发布频率使整包发布成为主要瓶颈。
- 存在明确的第三方适配器生态需求，并已具备签名、权限和沙箱方案。
- 本地功能需要独立升级、崩溃隔离或远程调用，且服务化收益可量化。
