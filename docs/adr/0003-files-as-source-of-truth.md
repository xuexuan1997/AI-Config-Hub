# ADR-0003：配置文件作为事实来源

## 状态

已接受，适用于 MVP 技术基线。

## 背景

AI Config Hub 管理的配置原本属于 Claude Code、Cursor、Codex、OpenCode 及用户项目。用户、Git、编辑器和目标工具都可能在产品之外修改这些文件。如果把 SQLite 中的规范化记录视为权威，系统会产生双重事实来源，并可能在部署时覆盖外部变更。

## 决策

- 原始配置文件始终是事实来源。
- SQLite 同时保存两类数据：可从配置文件重建的发现索引、规范化数据、引用关系和派生诊断；以及不能仅靠配置文件重建的用户设置、部署历史、任务审计和备份关联。
- 每次读取或写入决策都关联规范化路径、内容哈希、适配器版本和 Schema 版本。
- 文件监听触发增量重建；监听事件丢失或索引可信度不足时执行全量扫描。
- 部署预览记录源和目标哈希。执行前发现哈希变化时，中止部署并重新预览。
- 删除索引不会删除配置文件；删除或丢失配置文件后，下一次扫描将索引标记为缺失或移除。
- 数据库迁移失败时进入只读恢复状态，不用旧索引驱动写入。派生索引可以丢弃后重扫；非派生记录必须通过一致的数据库备份恢复，缺失时明确报告数据损失范围。

详细流程见[数据存储](../architecture/data-storage.md)与[可观测性与恢复](../operations/observability-and-recovery.md)。

## 备选方案

### SQLite 为权威存储并生成配置文件

中央模型更统一，但会改变用户现有工具和 Git 工作流，并使数据库损坏直接威胁原始资产。

### 中央 Git 仓库为唯一事实来源

适合团队发布规范，不适合发现和解释用户机器上已存在、尚未导入的多层级配置。

### 双向同步且双方均可覆盖

表面灵活，但冲突规则难以解释，容易在文件变化竞态中丢失数据。

## 后果

正面影响：

- 用户可继续使用编辑器、Git 和目标工具原生方式管理文件。
- SQLite 中的派生索引可在损坏或版本不兼容时安全重建，非派生审计和设置必须依靠数据库备份恢复。
- 漂移检测与部署确认有明确比较对象。

负面影响：

- 查询结果可能在文件变化后短暂过期，需要监听和任务状态提示。
- 规范化和生效计算必须可重复执行。
- 历史记录不能保证源文件仍然存在，必须保留足够的哈希和备份元数据，并纳入数据库一致备份。

## 复审条件

- 产品引入云端权威资产库或强制策略下发。
- 目标工具提供事务式配置 API，文件不再是主要载体。
- 团队治理要求改变现有本地优先模型。
