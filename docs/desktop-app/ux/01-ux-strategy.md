# 01 UX 策略

## 背景

AI Config Hub 桌面端已经具备本地扫描、资产解释、迁移预览、部署确认、历史和回滚能力。当前 UI 的主要问题不是能力缺失，而是能力之间的上下文联系不足：

- Overview 告诉用户有多少资产和历史，但没有说明下一步该处理什么。
- Assets 展示资产和诊断，但缺少优先级、筛选、批量处理和详情分层。
- Migration 能生成预览，但用户需要自行理解选择、兼容性、diff、确认和后续部署的关系。
- Deployment 保留安全确认，但确认项和它们对应的计划证据分离。
- History 展示记录，但不够突出恢复决策：是否验证、是否有备份、是否可回滚、当前是否被恢复锁阻断。

重设计应把桌面端定位成“本地配置治理工作台”，而不是多个独立功能页。

## 目标用户

### 个人开发者

个人开发者经常在 Claude Code、Cursor、Codex、OpenCode 之间切换。他们需要快速知道当前项目里有哪些配置、哪些真正生效、能否迁移到另一个工具，以及部署后如何撤销。

成功体验：

- 选择项目后能一键扫描。
- 扫描结果直接告诉他是否有错误、警告、冲突或可迁移资产。
- 迁移前能看到具体文件改动和字段损失。
- 部署后能看到验证结果和回滚入口。

### 团队配置维护者

团队维护者关心项目级配置一致性、跨工具迁移质量和可审计记录。他们需要诊断解释、计划证据、历史快照和可复现的回滚。

成功体验：

- 能按工具、资源类型和作用域定位问题。
- 能解释为什么某个资产被贡献、覆盖或忽略。
- 能保存或复制诊断和部署证据。
- 能识别哪些风险需要人工确认。

### 受限环境用户

这类用户可能在老旧 Linux、远程桌面、缺少系统文件选择器或受限权限环境中使用桌面端。他们需要明确的回退路径和不依赖云服务的本地操作。

成功体验：

- 文件选择器失败时，手动路径输入是清晰的备用路径。
- 权限不足、只读恢复和恢复锁都有具体说明。
- UI 不暗示会自动上传、执行脚本或修改未经确认的文件。

## 当前体验问题

### 1. 首次启动缺少一体化引导

当前空 Overview 里，项目选择位于顶部，扫描按钮位于内容区，两者视觉上分离。用户需要自己推断“选择项目”和“开始扫描”的关系。

改进方向：

- 首屏使用“项目上下文 + 只读扫描承诺 + 下一步”组合。
- 选择项目后，扫描 CTA 与项目路径保持同一区域。
- 无项目时禁用或弱化后续页面的破坏性操作，并解释原因。

### 2. 状态消息过于内部化

当前扫描成功消息可能暴露 `task:scan:*` 等内部标识。它对工程调试有价值，但不应成为主状态文案。

改进方向：

- 主文案面向用户：`Scan complete: 3 assets indexed, 1 issue needs review.`
- 技术 ID 放入详情展开区或诊断复制区。
- 所有异步任务使用相同结构：状态、进度、结果、下一步、技术详情。

### 3. 资产页缺少行动优先级

当前资产页按资源类型分组后较易扫描，但用户仍需要逐行查看。诊断计数没有转化为处理队列。

改进方向：

- 页面顶部显示健康摘要：Errors、Warnings、Partial conversions、Disabled、Ready to migrate。
- 默认排序：error > warning > disabled > recently changed > resource type。
- 表格支持筛选和搜索。
- 资产详情使用侧边检查器或 modal，但信息分层必须以行动为中心。

### 4. 迁移和部署之间缺少计划连续性

Migration 生成计划，Deployment 执行计划，但两个页面之间缺少显式的“当前计划”载体。用户到了 Deployment 页后，需要记住刚才选择了什么。

改进方向：

- 引入全局 Current plan 条。
- Migration 页面生成计划后，显示 `Ready for approval` 状态和 `Review deployment` CTA。
- Deployment 页面顶部必须复述计划摘要：来源、目标、操作数、兼容性、hash、过期时间、漂移状态。

### 5. 确认项没有指向证据

当前 Deployment 的确认 checkbox 是安全的，但它们像独立声明，缺少上下文。

改进方向：

- 每项确认旁展示对应证据链接，例如 `Overwrite existing target files` 指向 affected files。
- 部署总确认拆成“我已审阅 diff”“我理解会写入文件”“我理解部分转换字段损失”。
- 所有必需确认来自 plan 的 `requiredConfirmations`，UI 不额外发明不可追踪风险。

### 6. 历史页记录优先，恢复决策不足

历史记录展示了 kind、status、timestamp、snapshot 和 diff，但用户恢复时首先关心当前记录是否可作为回滚依据。

改进方向：

- 历史列表默认显示可回滚状态、验证状态、快照状态和影响文件数。
- 详情页顶部显示 recovery summary。
- 回滚入口与验证证据放在同一屏，失败或不可回滚时给出原因。

## 设计原则

### 1. 安全状态优先

任何可能写入磁盘的操作之前，UI 必须先展示：

- 当前计划是否来自最新扫描。
- 目标文件是否发生漂移。
- 会创建或替换哪些文件。
- 是否会丢弃字段或产生部分转换。
- 备份和验证如何执行。

### 2. 诊断可解释

诊断不是红黄数字。每条诊断必须能回答：

- 问题是什么？
- 来自哪个文件、哪个作用域、哪个资源？
- 影响哪个流程：扫描、解释、迁移、部署、回滚？
- 用户下一步能做什么？

### 3. 计划是核心对象

迁移预览、部署确认、历史记录和回滚都围绕计划生命周期组织：

```text
No plan
-> Draft selection
-> Preview generated
-> Needs confirmation
-> Ready to deploy
-> Deploying
-> Verified
-> Rollback available
```

### 4. 渐进披露

默认显示面向决策的摘要，允许用户深入查看完整证据：

- 默认：健康、风险、操作数、目标路径、下一步。
- 次级：字段损失、贡献者、覆盖关系、hash、diff。
- 详细：raw normalized payload、完整 task event、内部 ID。

### 5. 本地优先和可信边界可见

界面文案持续强化：

- 扫描默认只读。
- 不执行第三方脚本。
- 写入前一定预览。
- 部署可备份、可验证、可回滚。
- 配置文件仍是事实来源。

## 目标信息架构

保留现有左侧导航，但调整语义和页面职责。

| 导航 | 当前角色 | 重设计角色 |
| --- | --- | --- |
| Overview | 总览和扫描入口 | 项目健康中心、扫描入口、当前计划摘要 |
| Assets | 资产列表和详情 | 资产分诊、诊断队列、生效配置解释入口 |
| Migration | 迁移预览 | 计划构建器：选择来源、目标、兼容性、diff |
| Deployment | 执行部署和回滚按钮 | 计划审批和执行控制台 |
| History | 历史列表和详情 | 证据库、回滚入口、恢复状态 |
| Settings | 主题和语言 | 本地数据、备份、恢复、安全偏好、主题语言 |

全局上下文层：

- 项目路径和扫描状态。
- 当前计划状态。
- 当前异步任务状态。
- 恢复锁和只读恢复状态。

## 导航规则

- 没有项目时，Overview 为唯一主要路径，其他页面显示轻量空态和返回项目选择的 CTA。
- 没有扫描结果时，Assets、Migration 和 Deployment 不显示空白表格，而显示“先扫描”的空态。
- 生成迁移预览后，左侧 Deployment 导航项显示 `Needs approval` 徽标。
- 部署成功后，History 导航项显示 `New` 或最新记录摘要。
- 恢复锁存在时，Deployment 和 History 导航项都显示警示徽标，History 成为推荐入口。

## 成功指标

### 可用性指标

- 新用户能在 2 分钟内完成选择项目、扫描、打开资产详情。
- 已扫描项目中，用户能在 3 次点击内找到 error 诊断来源。
- 从迁移预览到部署确认，用户不需要返回 Migration 页面回忆目标文件。
- 部署完成后，用户能在 2 次点击内找到对应历史记录和回滚入口。

### 质量指标

- 主状态文案不暴露内部 task ID、plan ID 或 raw error，除非在详情区。
- 每个禁用按钮旁都有原因。
- 所有写入前确认都有计划证据链接。
- 空、加载、部分成功、失败、权限不足、恢复锁和只读恢复状态均有测试覆盖。

## 不变约束

- Renderer 仍不得直接访问文件系统、SQLite、Git 或 shell。
- 桌面端和 CLI 继续共享核心 API 和错误语义。
- 扫描默认只读，不执行外部配置。
- 部署必须来自新鲜预览计划，且保留 hash、diff、确认、备份、验证和历史记录。
- 回滚必须以可验证记录和备份为依据，不能成为盲写恢复。
