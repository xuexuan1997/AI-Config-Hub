# 2. 产品概述

AI Config Hub 是一款面向 AI Coding 工具的本地配置资产管理平台，用于统一发现、管理、诊断、转换和部署不同 AI IDE、AI CLI 以及自研 Agent 工具中的配置资源。

随着 AI Coding 工具快速发展，开发者通常会同时使用 Claude Code、Cursor、Codex、OpenCode 等多个产品。不同工具在模型、交互体验、代码编辑和任务执行方面各有优势，但它们普遍包含相似的配置概念，例如：

- Rules
    
- Agents
    
- Skills
    
- MCP Servers
    
- Commands
    
- Hooks
    
- Permissions
    
- Ignore 配置
    
- 工具 Settings
    

这些配置在概念上相似，但在目录结构、文件格式、配置层级、生效规则和功能语义上并不统一。

用户从一个工具切换到另一个工具时，往往需要重新复制、重命名、修改和验证配置。相同配置被复制到多个工具后，又会逐渐形成多个内容不一致的版本。

与此同时，同一工具中还可能同时存在用户级、项目级和子目录级配置。不同层级的文件混合在一起，使用户很难回答：

- 当前系统中到底存在哪些 AI Coding 配置？
    
- 某项配置来自家目录还是项目目录？
    
- 当前项目实际使用的是哪一份配置？
    
- 为什么修改了某个文件却没有生效？
    
- 某项配置是否被其他层级覆盖？
    
- 切换到另一个工具后哪些配置可以继续使用？
    
- 多个工具中的同名配置是否已经不一致？
    

AI Config Hub 通过统一资产模型和工具适配器，将不同工具中的配置转换为可管理、可追踪、可诊断和可迁移的配置资产。

产品不只是一个文件复制或同步工具，而是 AI Coding 配置资产的本地控制面。

# 3. 产品背景

## 3.1 多工具并存成为常态

不同 AI Coding 工具有各自适用的场景：

- Cursor 更接近集成式 AI IDE。
    
- Claude Code、Codex、OpenCode 更适合命令行、Agent 执行和自动化工作流。
    
- 企业内部还可能存在自研 Agent、IDE 插件或开发平台。
    

用户会根据任务类型、模型能力、成本、运行环境和团队要求在多个工具之间切换。

用户真正希望复用的通常不是某个工具本身，而是已经积累的：

- 编码规范
    
- 项目上下文
    
- Agent 定义
    
- Skill 工作流
    
- MCP 服务配置
    
- 常用命令
    
- 安全和权限约束
    

这些配置逐渐成为开发者的重要数字资产。

## 3.2 配置标准尚未统一

不同 AI Coding 工具使用不同的：

- 目录名称
    
- 文件名称
    
- 文件格式
    
- Frontmatter 字段
    
- JSON Schema
    
- 配置发现机制
    
- 配置优先级
    
- 项目继承规则
    
- Agent 和 Skill 定义方式
    
- MCP 配置格式
    

即使两个工具都支持 Rules、Agents 或 MCP，其具体语义也可能并不完全一致。

因此，不能简单通过复制目录实现稳定迁移。

## 3.3 配置层级导致排查困难

配置可能存在于：

- 用户家目录
    
- 工具全局目录
    
- 工作区目录
    
- 项目根目录
    
- 项目子目录
    
- Git 仓库中的团队目录
    
- 用户自定义目录
    

不同层级的配置可能发生：

- 继承
    
- 合并
    
- 覆盖
    
- 同名冲突
    
- 内容重复
    
- 部分字段失效
    

大多数工具只展示配置文件本身，不会完整解释最终实际生效的配置结果。

## 3.4 现有工具存在明显边界

当前一些开源产品已经能够解决部分问题，例如：

- 建立中央 Skill 库。
    
- 将 Skill 同步到多个工具。
    
- 从统一 Rules 源生成多个工具的配置。
    
- 在不同 AI 工具格式间进行转换。
    
- 使用 Git 管理配置版本。
    

但现有方案通常存在以下一种或多种限制：

- 主要管理 Skills 等单一资源类型。
    
- 更关注配置生成，而不是已有配置扫描。
    
- 缺少用户级、项目级和子目录级的统一层级视图。
    
- 无法解释最终生效配置。
    
- 缺少配置冲突和内容漂移诊断。
    
- 主要提供 CLI，普通用户理解成本较高。
    
- 桌面应用依赖较新的系统组件。
    
- 老旧 Linux 环境无法直接运行。
    
- 无法覆盖企业内的自定义目录和自研工具。
    
- 缺少安全的备份、预览和回滚机制。
    

AI Config Hub 将在已有开源产品能力的基础上，重点解决多资源、配置层级、生效诊断、跨工具迁移和老系统兼容问题。

# 4. 核心问题

## 4.1 配置分散

同一个用户的配置可能分布在多个工具和多个目录中，缺少统一入口。

用户很难完整掌握当前环境中存在的配置资产。

## 4.2 层级不透明

用户无法直观看出配置的来源、继承和覆盖关系。

当配置没有生效时，只能逐个目录排查。

## 4.3 工具切换成本高

从 Claude Code 切换到 Cursor、Codex 或 OpenCode 时，需要手工完成：

- 查找配置
    
- 判断哪些配置有价值
    
- 复制文件
    
- 修改目录
    
- 修改格式
    
- 删除不支持字段
    
- 验证新工具是否识别
    

该过程耗时且容易出错。

## 4.4 配置内容漂移

同一条 Rule 或同一个 Skill 被复制到多个工具后，各个副本独立修改，最终出现：

- 同名不同内容
    
- 版本无法确认
    
- 修复未同步
    
- 部分工具仍使用旧配置
    
- 无法判断哪一份是正确版本
    

## 4.5 配置实际效果无法确认

用户能够看到配置文件，但不容易确认：

- 工具是否真正识别了该文件。
    
- 当前目录是否在配置作用范围内。
    
- 是否被更高优先级配置覆盖。
    
- 是否存在语法错误。
    
- 是否使用了工具不支持的字段。
    
- Agent 引用的 Skill 是否存在。
    
- MCP 配置是否完整。
    

## 4.6 运行环境受限

部分企业和服务器环境仍使用较旧的 Linux 系统。

依赖新版本 WebKitGTK、glibc 或桌面组件的产品无法安装，导致配置管理工具本身无法覆盖真正需要它的受限环境。

# 5. 产品目标

## 5.1 总体目标

建立一套与具体 AI Coding 工具解耦的配置资产管理平台，实现：

- 配置统一发现
    
- 配置统一展示
    
- 配置层级分析
    
- 实际生效配置计算
    
- 配置健康诊断
    
- 跨工具配置转换
    
- 安全部署与回滚
    
- 多工具资产复用
    
- 多设备和团队共享
    
- 老旧 Linux 环境运行
    

## 5.2 用户价值

使用产品后，用户应能够通过统一界面或 CLI 完成：

```text
扫描当前环境
→ 查看配置分布
→ 选择工具和项目
→ 查看最终生效配置
→ 发现冲突或错误
→ 导入中央资产库
→ 转换为目标工具格式
→ 预览变更
→ 部署并验证
→ 必要时回滚
```

用户从“查找和复制配置文件”转变为“管理和部署配置资产”。

## 5.3 核心产品问题

产品必须能够清晰回答三个问题：

### 当前有什么配置？

展示所有工具、目录、项目和层级中的配置资源。

### 当前什么配置生效？

根据目标工具自身的配置发现和优先级规则，计算最终实际生效结果。

### 切换工具后会发生什么？

展示配置转换结果、兼容情况、差异、信息丢失和目标目录变化。

- 在线 Skill 市场。

- 产品自有用户账号系统。
    
- 组织和角色权限系统。
    
- 企业审批流。
    
- 服务端策略强制下发。
    
- 多人实时编辑。
    
- 内置完整代码编辑器。
    
- 自建密钥托管系统。

# 7. 目标用户

## 7.1 多工具开发者

同时使用 Claude Code、Cursor、Codex 或 OpenCode，希望减少配置重复维护成本。

核心需求：

- 多工具配置同步
    
- 配置迁移
    
- 配置差异查看
    
- 个人配置资产沉淀
    

## 7.2 AI Coding 重度用户

拥有大量 Rules、Agents、Skills 和 MCP 配置，需要统一分类和管理。

核心需求：

- 中央资产库
    
- 标签和搜索
    
- 场景化组合
    
- 版本管理
    
- 多项目部署
    

## 7.3 企业研发人员

工作在受限网络、旧版 Linux、远程服务器或内部研发环境中。

核心需求：

- glibc 2.28 兼容
    
- 无图形环境运行
    
- 离线安装
    
- 只读扫描
    
- 自定义数据目录
    
- 私有 Git 仓库
    
- 不上传配置内容
    

## 7.4 团队技术负责人

希望通过统一资产仓库维护团队的 AI Coding 规范和能力。

核心需求：

- Git 团队资产库
    
- 项目级配置
    
- 版本和变更记录
    
- 团队共享 Rules、Agents、Skills 和 MCP 模板
    
- 冲突提示
    

个人开发者和企业用户使用相同的产品核心和资产模型。

产品不为二者建设不同的功能体系，而是通过部署方式、网络策略、只读模式和 Git 仓库类型满足不同场景。

# 8. 产品设计原则

## 8.1 核心引擎与界面解耦

扫描、解析、转换、诊断、部署和回滚能力必须位于统一核心引擎中。

CLI、本地 Web UI 和桌面应用只是不同入口，不得分别实现独立业务逻辑。

## 8.2 工具差异显式保留

统一资产模型不代表强行将所有工具配置完全抹平。

产品需要同时保留：

- 通用字段
    
- 工具专属字段
    
- 原始配置
    
- 转换后的配置
    
- 不兼容信息
    

## 8.3 读取优先于写入

产品首次扫描默认只读，不应自动移动、删除或重写现有配置。

所有写入操作必须经过：

- 冲突检查
    
- Dry Run
    
- 差异预览
    
- 用户确认
    
- 原始文件备份
    
- 写入后验证
    

## 8.4 诊断必须可解释

系统不得只告诉用户“存在冲突”或“配置未生效”。

诊断需要展示：

- 问题文件
    
- 来源层级
    
- 影响范围
    
- 判断依据
    
- 覆盖关系
    
- 建议处理方式
    

## 8.5 不执行第三方配置

扫描 Skill、MCP、Hook 或其他配置时，默认只读取文本，不执行其中的脚本或命令。

## 8.6 优先兼容受限环境

核心 CLI 和服务不得依赖桌面图形组件，最低支持 glibc 2.28。

# 9. 竞品分析

## 9.1 Skills Manager 类产品

此类产品通常围绕中央 Skill 库展开，支持：

- Skill 导入
    
- 多工具同步
    
- 项目工作区
    
- Preset
    
- 内容哈希
    
- Git 版本管理
    
- GUI 和 CLI
    

优势：

- Skill 管理体验完整。
    
- 中央库和项目部署模型清晰。
    
- 能够减少 Skill 重复复制。
    

局限：

- 产品模型主要围绕 Skill。
    
- Rules、Agents、MCP 等资源管理不足。
    
- 缺少完整的配置层级分析。
    
- 无法回答指定目录最终生效哪些配置。
    
- 部分桌面技术栈对旧 Linux 系统要求较高。
    

## 9.2 规则生成类产品

此类产品通常维护一份统一规则源，再为不同工具生成对应格式。

优势：

- 配置源相对统一。
    
- 支持多个目标工具。
    
- 适合在新项目中建立规范。
    

局限：

- 更强调生成，而不是发现已有配置。
    
- 需要用户主动迁移到规定目录。
    
- 对历史项目和现有环境扫描不足。
    
- 对配置实际生效结果解释不足。
    

## 9.3 跨工具转换类产品

此类产品支持从一个工具导入配置并输出到另一个工具。

优势：

- 工具覆盖范围较广。
    
- 支持多种资源。
    
- 适合一次性迁移。
    

局限：

- 通常以 CLI 命令和一次性转换为主。
    
- 缺少长期资产状态管理。
    
- 缺少多层级配置视图。
    
- 缺少持续漂移检测和回滚管理。
    

## 9.4 本产品差异化

AI Config Hub 的核心差异不在于支持更多复制目标，而在于同时提供：

1. **发现**：扫描已有配置环境。
    
2. **建模**：统一管理多种配置资源。
    
3. **解释**：计算实际生效配置。
    
4. **诊断**：定位冲突、覆盖和漂移问题。
    
5. **转换**：在不同工具间进行兼容性可见的转换。
    
6. **部署**：安全写入目标工具目录。
    
7. **治理**：通过 Git 进行版本管理和团队共享。
    
8. **兼容**：支持 glibc 2.28 和无桌面 Linux 环境。

# 11. 核心概念

## 11.1 Tool

代表目标 AI Coding 工具。

例如：

- Claude Code
    
- Cursor
    
- Codex
    
- OpenCode
    

## 11.2 Resource

代表一种工具配置资源。

MVP 中包括：

- Rule
    
- Agent
    
- Skill
    
- MCP
    

## 11.3 Scope

代表配置的生效范围和层级。

统一模型包含：

- Personal：个人中央资产
    
- Team：团队中央资产
    
- Global：工具用户级配置
    
- Workspace：工作区配置
    
- Project：项目级配置
    
- Directory：项目子目录配置
    
- Session：工具临时会话配置
    

不同工具并不一定支持全部 Scope。

## 11.4 Asset

经过统一管理的配置资源。

每项资产至少包含：

- 唯一 ID
    
- 名称
    
- 资源类型
    
- 描述
    
- 标准化内容
    
- 原始内容
    
- 来源工具
    
- 来源路径
    
- 来源层级
    
- 适用工具
    
- 兼容等级
    
- 标签
    
- 内容哈希
    
- 版本
    
- 依赖关系
    
- 创建时间
    
- 更新时间
    

## 11.5 Adapter

负责连接统一资产模型和具体工具格式。

适配器包括：

- Detector：工具和目录识别
    
- Scanner：配置扫描
    
- Parser：原生格式解析
    
- Scope Resolver：层级识别
    
- Effective Resolver：生效配置计算
    
- Transformer：格式转换
    
- Renderer：目标文件生成
    
- Validator：配置校验
    

## 11.6 Effective Configuration

针对指定工具、项目和目录，根据该工具真实规则计算出的最终配置集合。

Effective Configuration 不只包含最终内容，还需要包含：

- 来源
    
- 加载顺序
    
- 继承关系
    
- 覆盖关系
    
- 冲突信息
    
- 不兼容字段
    
- 判断依据
    

## 11.7 Deployment

将中央资产转换并写入目标工具目录的过程。

部署方式包括：

- Copy
    
- Symlink
    
- Hardlink
    
- Generated File

# 12. 统一资产模型

## 12.1 资产基本结构

示例：

```yaml
schema_version: 1

id: rule-code-review
name: Code Review Rules
type: rule
description: 代码审查规范

source:
  type: imported
  tool: claude-code
  path: /workspace/project/.claude/rules/code-review.md
  scope: project

content:
  format: markdown
  body: |
    审查代码时重点关注安全性、可维护性和兼容性。

extensions:
  claude-code: {}
  cursor: {}
  codex: {}
  opencode: {}

compatibility:
  claude-code: full
  cursor: full
  codex: partial
  opencode: full

tags:
  - review
  - quality

version: 1
hash: sha256:xxx
created_at: 2026-06-21T10:00:00Z
updated_at: 2026-06-21T10:00:00Z
```

## 12.2 原始内容保留

系统导入工具配置时，必须保留原始内容。

原因包括：

- 不同工具字段不能完全映射。
    
- 未来适配器升级后可能支持更多字段。
    
- 用户需要查看转换前后差异。
    
- 回滚时需要恢复原始文件。
    

## 12.3 兼容等级

资源转换结果分为：

|兼容等级|含义|
|---|---|
|Full|可以完整转换，语义基本保持一致|
|Equivalent|表达形式不同，但可以转换为等价能力|
|Partial|只有部分字段或能力可以转换|
|Degraded|可以生成目标配置，但功能会降级|
|Manual|需要用户手工处理|
|Unsupported|目标工具不支持|

# 15.8 配置诊断

系统需要识别以下问题。

## 文件和格式问题

- 文件无法读取
    
- 编码异常
    
- JSON 语法错误
    
- JSONC 语法错误
    
- YAML 语法错误
    
- TOML 语法错误
    
- Markdown Frontmatter 错误
    
- Schema 字段错误
    

## 目录问题

- 目录位置不被工具识别
    
- 文件名称不符合工具要求
    
- 资源放在错误目录
    
- 符号链接失效
    
- 符号链接循环
    
- 目录权限不足
    

## 内容问题

- 同名不同内容
    
- 相同内容重复
    
- 配置为空
    
- 必填字段缺失
    
- 使用工具不支持字段
    
- Agent 引用不存在的 Skill
    
- Skill 引用资源不存在
    
- MCP 环境变量缺失
    

## 层级问题

- 配置被其他层级覆盖
    
- 配置处于作用范围之外
    
- 多个层级定义冲突
    
- 用户认为配置生效但实际被忽略
    
- 全局配置和项目配置语义矛盾
    

## 同步问题

- 中央资产更新但目标副本未更新
    
- 目标副本被手工修改
    
- 中央资产和目标副本同时修改
    
- Git 工作区存在未提交修改
    
- Git 远程版本更新
    
- 内容哈希不一致
    

## 安全问题

- MCP 中存在疑似明文密钥
    
- 配置中包含 Token
    
- 配置中包含 Password
    
- Skill 中包含可执行脚本
    
- MCP Command 指向未知程序
    
- 配置引用项目外路径
    

诊断严重程度：

|等级|说明|
|---|---|
|Error|配置无法被正常使用，或存在明显安全问题|
|Warning|配置可能失效、被覆盖或存在兼容风险|
|Notice|存在重复、漂移或可以优化的问题|

每条诊断必须包含：

- 问题描述
    
- 相关资源
    
- 来源文件
    
- 影响范围
    
- 判断依据
    
- 建议处理方式

# 15.13 同步状态

中央资产和部署副本之间的状态包括：

|状态|含义|
|---|---|
|Not Deployed|尚未部署|
|In Sync|内容一致|
|Center Newer|中央资产发生更新|
|Target Newer|目标副本被修改|

|Missing|目标文件丢失|
|Invalid|目标文件无法解析|
|Unsupported|当前工具不支持|

- Authorization
    
- Password

- Cookie
    
- Credential


```yaml
env:
  API_TOKEN: ${API_TOKEN}
```

MVP 不保存和管理环境变量的真实值，也不建设密钥保险库。

产品不负责启动和停止 MCP Server。

# 23. MVP 验收标准

## 23.1 工具发现

在安装或使用过 Claude Code、Cursor、Codex 和 OpenCode 的环境中，系统能够识别对应工具和配置目录。

无法识别时，需要展示原因，并允许手工配置目录。

## 23.2 多层级扫描

给定包含用户级、项目级和子目录级配置的环境，系统能够：

- 找到配置文件
    
- 判断资源类型
    
- 判断所属工具
    
- 判断所属 Scope
    
- 展示来源路径
    

不得静默忽略无法解析的文件。

## 23.3 生效配置

用户选择一个工具、项目和子目录后，系统能够展示：

- 最终生效资源
    
- 配置来源
    
- 加载顺序
    
- 覆盖关系
    
- 未生效原因
    

## 23.4 配置诊断

当存在以下问题时，系统能够给出明确诊断：

- 同名不同内容
    
- 配置格式错误
    
- 配置被覆盖
    
- 失效链接
    
- Agent 引用缺失
    
- MCP 敏感字段
    
- 中央资产和目标副本不一致
    

## 23.5 跨工具迁移

用户能够将源工具中的 Rules、Agents、Skills 和 MCP 迁移到目标工具。

系统需要在执行前展示：

- 转换结果
    
- 兼容等级
    
- 丢失字段
    
- 目标文件路径
    
- 文件差异
    

## 23.6 安全部署

部署失败时：

- 原文件不能损坏。
    
- 临时文件需要清理。
    
- 用户能够查看失败原因。
    
- 用户能够恢复部署前状态。
    

## 23.7 Git 团队资产

两名用户可以通过同一个 Git 仓库共享资产。

系统能够：

- Pull 更新
    
- 显示本地变更
    
- Commit 和 Push
    
- 提示 Git 冲突
    

不得在冲突时自动覆盖任意一方。

## 23.8 外部编辑器

用户从资产详情打开文件后，外部编辑器保存变更，产品能够重新扫描并更新状态。

## 23.9 只读模式

只读模式运行期间：

- 不修改被扫描文件。
    
- 不创建链接。
    
- 不修改 Git 工作区。
    
- 不启动 MCP Server。
    
- 不执行 Skill 脚本。
    
- 不在项目目录写入缓存。
    

## 23.10 系统兼容

# 25. 产品风险

## 25.1 工具规则变化

Claude Code、Cursor、Codex 和 OpenCode 可能调整目录、格式和优先级规则。

应对方式：

- 适配器版本化
    
- 工具规则与核心引擎解耦
    
- 建立兼容测试
    
- 标记未经验证的工具版本
    
- 保留原始配置
    

## 25.2 配置语义无法完全转换

不同工具即使使用相同资源名称，也可能不具备等价能力。

应对方式：

- 明确兼容等级
    
- 不承诺全部无损转换
    
- 展示字段丢失
    
- 允许外部编辑器调整
    
- 保存转换报告
    

## 25.3 生效配置判断错误

如果优先级和发现规则实现不准确，诊断结果会误导用户。

应对方式：

- 将适配器测试优先于 UI 开发
    
- 为每个工具建立真实目录样本
    
- 提供判断依据
    
- 允许用户报告适配问题
    
- 在不确定时明确标记，而不是给出确定结论
    

## 25.4 双向同步覆盖数据

中央资产和工具目录可能同时修改。

应对方式：

- 使用内容哈希
    
- 保存同步基线
    
- 识别 Diverged
    
- 禁止自动覆盖
    
- 要求用户确认合并策略
    

## 25.5 MCP 和 Skill 安全风险

配置中可能包含密钥或可执行命令。

应对方式：

- 默认不执行
    
- 敏感字段脱敏
    
- 风险标记
    
- 写入日志过滤
    
- 只读模式
    
- 外部命令确认
    

## 25.6 老系统兼容限制

部分依赖可能隐式要求更高版本 glibc。

应对方式：

- 使用固定构建环境
    
- 在 glibc 2.28 环境持续测试
    
- 检查二进制依赖
    
- Core Engine 和桌面壳分开发布
    
- 提供 CLI 和本地 Web UI 作为稳定入口
    

## 25.7 MVP 范围过大

同时支持四个工具和四种资源会增加适配复杂度。

应对方式：

- 先实现扫描和只读诊断
    
- 分阶段开放写入能力
    
- 每个适配器按资源类型逐步达到完整支持
    
- 在 UI 中展示能力矩阵
    
- 不因追求工具数量牺牲结论准确性

# 28. 产品总结

AI Config Hub 解决的不是单个 AI IDE 的配置问题，而是随着 AI Coding 工具增多而出现的配置资产碎片化问题。

其核心价值可以概括为：

```text
统一发现配置
+ 看清配置层级
+ 解释实际生效结果
+ 诊断冲突和漂移
+ 跨工具安全迁移
+ 多工具资产复用
+ Git 版本和团队共享
+ 老旧系统兼容
```

产品最终希望让用户能够自由选择和切换 AI Coding 工具，而不需要反复重建已经积累的 Rules、Agents、Skills 和 MCP 配置资产。
